# ADR-0002：采用 Documents 托管工作区

## 状态

Accepted — 2026-07-11

## 背景

应用此前以导入单个 C 文件为入口，启动时还会自动载入演示代码。它无法表达持续项目、临时实验和测试材料，也没有一个用户可见、可恢复的本地事实源。新的 Dashboard 需要像文档管理器一样列出项目、沙箱和测试；新建条目必须立即落到用户的 Documents，并在编辑时持续同步。

同时，renderer 仍不能获得任意文件系统能力或绝对路径。直接把路径传给页面、允许页面拼接路径，或把所有条目塞进 `localStorage`，都会破坏现有 Electron 安全边界或数据可恢复性。

## 决策

- 生产环境使用 `~/Documents/C Algorithm Workbench` 作为托管根目录，并固定建立 `Projects`、`Sandboxes`、`Tests` 三个分类目录。
- 每个条目使用不可预测的 `<kind>-<uuid>` 独立子目录，包含版本化 `entry.json` 和唯一源码事实源 `main.c`。
- renderer 只接收条目摘要、源码和 opaque ID；绝对路径不跨越 preload 边界。
- Electron 主进程严格验证 IPC 请求、条目 ID、标题、UTF-8 源码、大小和真实普通文件，并拒绝符号链接路径。
- 新建条目先写入最小 C17 程序，然后进入搭建工作区。
- 编辑后的源码经 300 ms 防抖串行保存；保存请求携带期望 revision，检测到磁盘版本冲突时拒绝覆盖。
- 保存失败后保留最新 dirty 源码，后续 flush 可重试；切换文档、刷新或关闭窗口必须先完成 flush。
- revision 冲突提供显式磁盘重载入口；只有用户确认且磁盘版本读取成功后，才放弃本地 dirty 源码。
- Electron 主进程在关闭窗口前向 renderer 发出具名请求，只有 renderer 确认工作区已保存后才允许关闭；失败响应保持窗口打开。
- `main.c` 与 manifest 使用同目录临时文件、刷盘和原子重命名更新。失败必须返回具名错误，不伪造“已保存”。
- Dashboard 只把托管条目视为自动保存文档。通过文件选择、拖放或粘贴打开的源码保持临时状态，不静默改写来源文件。
- 首版不暴露文件条目删除、重命名、外部文件监听和同步服务；引入这些能力前需要单独设计恢复、冲突和权限语义。

测试可通过 `PANEL_WORKSPACE_ROOT` 将根目录替换为临时目录；生产默认值不受 renderer 输入控制。

## 备选方案

### 方案 A：继续只导入任意 C 文件

- 优点：无需维护项目索引。
- 缺点：无法提供 Dashboard、条目类型和可靠自动保存；用户仍需自行整理散落文件。
- 放弃原因：不满足持续算法学习和设计工作台的目标。

### 方案 B：全部保存在 Electron userData 或 localStorage

- 优点：实现简单，应用内部定位方便。
- 缺点：用户难以在 Documents 中直接找到和备份源码，浏览器存储也不适合作为课程项目事实源。
- 放弃原因：与用户要求的本机专属 Documents 文件夹冲突。

### 方案 C：renderer 直接使用文件系统路径

- 优点：页面可以自行实现文件管理。
- 缺点：扩大攻击面，破坏窄 preload API，并容易产生路径穿越、符号链接和竞态问题。
- 放弃原因：安全代价不必要。

## 后果

### 正面

- 项目、沙箱和测试拥有统一但隔离的本地持久化模型。
- Dashboard 列表可由磁盘重建，应用重启不会依赖隐藏的浏览器状态。
- 自动保存状态、revision 冲突和失败原因可见，不会静默覆盖较新的磁盘版本。
- 目录模型可在未来扩展多文件、测试用例、课程元数据和版本历史。

### 负面

- 当前每个条目只有 `main.c`，多文件项目仍需后续 schema 迁移。
- 外部编辑器修改不会实时反向进入应用。
- 保存持续失败或 revision 冲突时，正常关闭会被阻止，用户必须先处理未保存状态或显式强制退出。

## 架构适应度检查

- 工作区 store 测试覆盖真实目录创建、列表、打开、保存、revision 冲突、大小限制、恶意 ID 和符号链接拒绝。
- preload 合同测试锁定具名工作区 API，Electron E2E 继续验证 renderer 无 Node、无绝对路径能力。
- Dashboard E2E 在临时根目录中新建真实条目、编辑 `main.c`、等待自动保存并在刷新后重新打开。
- 关闭握手 E2E 在 300 ms 防抖尚未触发时立刻关闭应用，再从磁盘核对最后一次源码修改。
- 冲突恢复 E2E 外部推进 manifest revision，验证界面拒绝覆盖并在确认后重新载入磁盘源码。
- 单元测试验证防抖合并、串行 revision 更新、切换条目前 flush 和失败状态呈现。
